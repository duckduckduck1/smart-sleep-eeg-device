# Прошивка

Подробности по прошивке STM32F103: как собирается, распиновка, обмен с
приложением оператора и особенности, о которых стоит знать заранее.

Общее описание проекта — в [README.md](../README.md), платы —
в [hardware.md](hardware.md).

## Как собирается

Два инструмента, у каждого своя роль:

1. **STM32CubeMX** — назначение ног и периферии. По файлу `EEG_sample.ioc` он
   генерирует код инициализации на HAL: `Core/Src/gpio.c`, `spi.c`, `tim.c`,
   `usart.c`, `dma.c` и каркас `main.c`.
2. **VisualGDB в Visual Studio** — сборка и прошивка. Сгенерированный проект
   импортируется сюда, и дальше вся работа идёт в нём.

Решение — `firmware/visualgdb/smart-sleep-eeg-device.sln`. Проект собирает
исходники из `firmware/Core/`, поднимаясь на два уровня вверх от своего файла,
плюс лежащий рядом драйвер `max30003.c`. Прошивка заливается по SWD
(ноги PA13/PA14).

> **STM32CubeIDE в работе не используется.** Файлы `.cproject`, `.project`,
> `.settings` и `.mxproject` — побочный продукт CubeMX: он создаёт их, потому
> что в качестве тулчейна выбран CubeIDE. Собрать по ним проект не получится
> (драйвер MAX30003 не входит в их пути включения), но это ничего не значит —
> ими никто не пользуется. Удалять их тоже незачем: CubeMX создаст их заново.

## Распиновка

| нога | назначение |
|---|---|
| PA5 | SPI1_SCK — MAX30003 |
| PA6 | SPI1_MISO — MAX30003 |
| PA7 | SPI1_MOSI — MAX30003 |
| PA4 | CS для MAX30003 — обрамляет транзакции макросами `NSS_down` / `NSS_up` |
| PA9 | USART1_TX — в JDY-16 |
| PA10 | USART1_RX — из JDY-16 |
| PA1 | TIM2_CH2 — **FCLK**, тактирование MAX30003, включается макросом `FCLK_on` |
| PA2 | TIM2_CH3 — ШИМ светодиода ФБМ (**в `.ioc` не описана**) |
| PA13 / PA14 | SWDIO / SWCLK — отладка и прошивка |

Периферия:

- **SPI1** — мастер, 8 бит, CPOL 0 / CPHA 0, NSS программный (нога PA4).
- **USART1** — 9600 бод, 8N1, приём по прерыванию, по одному байту.
- **TIM2** — предделитель 0, период 249. Заняты **оба** канала: канал 2 даёт
  тактовый сигнал MAX30003, канал 3 — ШИМ светодиода.
- **TIM3** — предделитель 8, период 1000.

Макросы `FCLK_on` / `FCLK_off` и `NSS_up` / `NSS_down` объявлены
в `max30003.h` и разворачиваются в работу с TIM2 и ногой PA4. По имени ноги
её назначение не найти — искать надо по макросу.

## Обмен с приложением

### Поток отсчётов

250 Гц, один отсчёт — три байта. Прошивка копит 12 отсчётов и отдаёт их одним
пакетом в 36 байт в UART, откуда их забирает JDY-16 и отправляет по BLE.

Формат отсчёта и его декодирование в микровольты описаны на стороне приложения
и здесь не дублируются:
[docs/data-format.md](https://github.com/duckduckduck1/smart-sleep-eeg-app/blob/main/docs/data-format.md).

> **Запас по скорости почти исчерпан.** 250 Гц × 3 байта = 750 Б/с, а UART на
> 9600 бод даёт 960 Б/с — занято около 78 %. Добавить в поток что-то ещё без
> повышения скорости не выйдет.

### Команда управления светом

Приходит по тому же UART, четыре байта:

```text
[on_off, pwm, 0x0D, 0x0A]
```

| байт | значение |
|---|---|
| `on_off` | `0x01` — включить ШИМ, `0x00` — выключить |
| `pwm` | скважность, **0…249** |
| `0x0D 0x0A` | обязательный конец пакета |

Разбирает команду `HAL_UART_RxCpltCallback` в `Core/Src/main.c`: байты копятся
до пары `0x0D 0x0A`, после чего пакет длиной ровно 4 байта принимается
к исполнению, остальные отбрасываются.

## Особенности

### Шкала ШИМ не совпадает с приложением

Период TIM2 равен 249, то есть полная яркость — это `pwm = 249`, и всё, что
больше, прошивка обрезает до 249. Приложение же считает
`pwm = round(уровень × 255 / 100)`.

В результате уровни 98 % и 99 % дают ту же яркость, что и ~97,6 %. Расхождение
небольшое и на работу не влияет — **исправлять его не планируется**, но при
разборе данных стоит помнить: яркость выше ~97 % не различается.

### Не перегенерировать код из `.ioc`

Часть логики дописана руками в файлах, которыми владеет CubeMX, и лежит вне
секций `USER CODE`:

| файл | что пропадёт при генерации |
|---|---|
| `Core/Src/main.c` | приём команд, настройка MAX30003, цикл чтения отсчётов, `SetDuty()` — маркеры `USER CODE` из файла удалены, осталась одна секция в самом конце |
| `Core/Src/tim.c` | настройка TIM2 канала 3 и инициализация ноги PA2 — вставлены между сгенерированными блоками |

Ноги давно назначены и переназначать их не нужно, так что на практике это не
мешает. Если всё же понадобится — сначала сохраните `main.c` и `tim.c`, потом
верните свои куски обратно.

### `.ioc` знает не про все ноги

В `.ioc` описан канал 2 TIM2 на ноге PA1 — это FCLK, и он верен. А канала 3 на
ноге PA2, который даёт ШИМ светодиода, там нет: он дописан руками прямо
в `tim.c`. Источник правды по ШИМ — код, не `.ioc`.

### Драйвер MAX30003 лежит внутри проекта VisualGDB

`max30003.c` и `max30003.h` лежат в `firmware/visualgdb/smart-sleep-eeg-device/`
— рядом с файлом проекта, а не вместе с остальными исходниками в `Core/`.
Из-за этого кажется, что проектов два. Проект **один**: `.vcxproj` собирает
и `Core/`, и драйвер.

Переносить драйвер в `Core/` никто не мешает, но это правка путей в файле
проекта, то есть риск сломать сборку — а работающую прошивку трогать без
надобности не стоит.
